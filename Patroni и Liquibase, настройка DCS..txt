=================== Настройка и администрирование кластеров Patroni:
это процесс создания, управления и поддержки высокодоступного кластера СУБД PostgreSQL, который автоматически обеспечивает отказоустойчивость и переключение на резервный узел (failover/switchover) при сбое, используя репликацию и распределенное хранилище конфигурации (DCS), такое как etcd, Consul, ZooKeeper, Kubernetes. 
Patroni 				.это автоматическая система аварийного переключения для PostgreSQL. 
Patroni 				.обеспечивает автоматическое и ручное переключение при отказе и хранит все важные данные в распределенном хранилище конфигурации (DCS) на базе систем ETCD, Consul и т.д.
Patroni 				.автоматизирует сложные задачи управления кластером, позволяя легко масштабировать его и поддерживать консистентность данных. Основные задачи при настройке/администрировании:
Развертывание: 				.Установка Patroni и PostgreSQL на нескольких узлах, настройка DCS.
Конфигурация: 				.Определение параметров кластера, репликации и DCS в конфигурационных файлах Patroni, например, для https://patroni.readthedocs.io/.
Автоматизация Failover/Switchover: 	.Patroni мониторит состояние узлов и при сбое основного (master) автоматически выбирает нового лидера, перенаправляя трафик.
Мониторинг: 				.Отслеживание состояния узлов, реплик и DCS для обеспечения здоровья кластера.
Управление репликацией: 		.Добавление новых реплик, изменение их типа (синхронная/асинхронная).
Интеграция с балансировщиками: 		.Настройка HAProxy или PgBouncer для маршрутизации запросов к активному узлу.
Динамическое конфигурирование: 		.Возможность изменять настройки PostgreSQL без перезапуска всего кластера. 
В итоге, это комплекс действий по созданию надежной и отказоустойчивой базы данных PostgreSQL с минимальным участием администратора для рутинных операций. 
==================== DCS в контексте PostgreSQL:
Distributed Configuration Store 	.распределенное хранилище конфигурации. Это внешняя система (например, etcd, Consul, ZooKeeper), которая используется фреймворками высокой доступности (типа Patroni) для координации узлов кластера, хранения их состояния, обеспечения консенсусуса и автоматического переключения при отказах, чтобы PostgreSQL кластер оставался доступным. 
Distributed Configuration Store		.Это надежное хранилище «ключ-значение», которое гарантирует, что все узлы кластера знают актуальный статус друг друга и могут выбрать лидера (мастера) без человеческого вмешательства. 
==================== Liquibase:
 — это open-source решение для управления схемами и миграциями баз данных. Оно помогает разработчикам и администраторам баз данных поддерживать синхронизацию структуры базы данных с кодом приложения. 
Liquibase 				.поддерживает множество систем управления базами данных: PostgreSQL, MySQL, Oracle, SQL Server и других. 
-------------------- Некоторые особенности Liquibase:
Версионность и контроль изменений:		.Каждое изменение схемы базы данных документируется и отслеживается, есть возможность отката к предыдущим версиям структуры.
Кроссплатформенность:				.Один и тот же changelog работает на разных СУБД.
Безопасность и согласованность:			.Автоматическая проверка применимости миграций, механизмы блокировки от параллельного применения конфликтующих изменений, встроенные транзакционные механизмы защиты целостности данных.
Интеграция с CI/CD:				.Лёгкая интеграция в pipeline непрерывной разработки, автоматизация процессов развёртывания базы данных.
Поддержка различных форматов описания миграций:	.xml, .yaml, .json, .sql.
Гибкость и расширяемость:			.Поддержка сложных сценариев миграции, возможность написания собственных расширений, поддержка контекстов и метаданных для миграций.
Генерация changelog				.из существующей БД (команда liquibase generateChangeLog) — полезно при переходе на Liquibase.
Context и labels				.можно применять миграции в нужных средах и при определённых параметрах, которые будут передаваться при запуске (например, для заполнения тестовых данных).
Предикаты (preConditions):			.можно проверить условия перед выполнением миграций.
Шаблоны (customChange):				.если нужно, можно реализовать собственные Java-классы для сложных миграций.
При первом запуске миграции Liquibase создаёт таблицу в схеме базы данных для отслеживания применённых изменений и дальше работает с ней автоматически. Если изменение уже применялось, повторного выполнения не будет.
Liquibase 				.автоматически создаёт и использует несколько служебных таблиц для отслеживания состояния миграций в базе данных, например, DATABASECHANGELOG — хранит информацию о выполненных миграциях, DATABASECHANGELOGLOCK — предотвращает одновременное выполнение миграций разными процессами.
----------------------- Принцип работы:
Изменения структуры базы данных записываются в файлы changelog. Эти файлы могут быть в одном из форматов: XML, YAML, JSON или SQL. Файлы changelog могут включать друг в друга другие файлы, что облегчает управление изменениями при большом объёме данных.
